iLink 职责 vs 我必须自己做的职责

iLink 是「管道」，不是「管家」

🦞 iLink（小龙虾）做这些 已具备

微信官方协议，只管"消息怎么到微信用户手里"

  • ✓
    个人微信用户能加 bot 好友
    扫码登录 → 获得 bot_token
  • ✓
    接收用户发来的消息
    长轮询 /getupdates 拉取
  • ✓
    回复用户消息
    /sendmessage + context_token
  • ✓
    消息类型支持
    文本/图片/文件/链接卡片
  • ✓
    识别用户身份
    每条消息带 user_id

🛠️ 我必须自己做的 全栈自建

用户管理、订阅、策略、匹配、推送编排——iLink 一概不管

  • ✗
    用户/企业档案管理
    哪个微信用户对应哪家企业，资质标签、关注行业
  • ✗
    订阅配置
    关注哪些政策类型、地区、关键词增强
  • ✗
    匹配规则 + 阈值
    多少分推送、高优先级阈值、个性化过滤
  • ✗
    推送频率 + 时间窗
    实时/日报/周报、避开夜间、不要刷屏
  • ✗
    推送配额/限流
    避免触犯微信风控（单用户每天 ≤ N 条）
  • ✗
    用户行为日志
    看了/没看/回复了/咨询了 → 用于优化匹配
  • ✗
    Web 管理后台
    让用户自己改配置、查看历史

MVP 现状 vs 完整方案

模块第一期 MVP完整 SaaS 需要状态
爬虫 3 个源配置（云上可跑） 12+ 源 / 反爬 / 失败告警 MVP ✓
LLM 摘要 MiniMax M3 · 单条 + 批量 摘要 + 匹配 + 详情解读 三模型协同 MVP ✓
用户管理 无（单租户 demo） users / companies / 资质 / 标签 需第二期
订阅配置 无（硬编码 mock） 每用户独立：行业/地区/类型/关键词 需第二期
匹配引擎 无（无用户也无匹配） 规则预筛 + Qwen 深度匹配 + 个性化 需第二期
推送编排 手动触发 1 条 定时 + 频率 + 限流 + 时间窗 + 去重 需第二期
微信通道 Mock 微信（无 iLink） iLink 客户端（扫码 + 长轮询） 部署后补
管理后台 触发页（demo 风格） Vue 3 SPA · 全功能 MVP ✓
事件日志 / 漏斗 push_logs 表 推送→查看→咨询 全链路追踪 MVP ✓
📌 一句话总结： iLink 解决"消息到微信"，我的服务解决"给每个用户推什么、什么时机推、推几次"——后者才是政策雷达真正的产品壁垒，iLink 一行代码也帮不了。
